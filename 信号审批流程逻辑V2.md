# 信号审批流程逻辑 V2

## 概述

信号的创建、编辑、端点增删操作涉及两阶段审批：完善阶段（completion）和审批阶段（approval）。V2 版本根据操作类型智能决定是否需要完善阶段。

## 操作类型与审批流程

| 操作 | 完善阶段 | 审批阶段 | 说明 |
|------|---------|---------|------|
| 新建信号 | 其他端点设备负责人完善确认 | 总体组审批 | 创建人可能不了解对端设备信息 |
| 添加新端点 | 仅新端点的设备负责人完善确认 | 总体组审批 | 已确认端点不需要重新确认 |
| 修改信号属性（端点已全部确认） | 跳过 | 总体组审批 | 信号属性不涉及端点信息 |
| 修改信号属性（有未确认端点） | 不允许提交 | — | 先完善端点再改属性 |
| 删除端点 | 跳过（知会被删端点负责人） | 总体组审批 | 总体组裁决拓扑变更 |

## 详细规则

### 1. 新建信号

- 创建人填写信号属性和端点信息
- 创建人自己负责的设备端点自动标记为已确认
- 其他端点的设备负责人收到"待完善"通知，需查看并确认/补充端点信息
- 所有端点确认后，自动进入审批阶段
- 总体组成员收到"待审批"通知

### 2. 已有信号添加新端点

- 仅新端点的设备负责人收到"待完善"通知
- 已有的已确认端点的设备负责人不需要参与
- 新端点确认后进入总体组审批

### 3. 修改信号属性

- **前提条件**：信号的所有端点必须已确认，否则不允许提交修改
- 修改信号属性（导线线规、敷设代码、连接类型等）不涉及端点信息
- 直接跳过完善阶段，提交总体组审批
- 其他端点设备负责人不需要参与

### 4. 删除端点

- 被删除端点的设备负责人收到知会通知（非审批）
- 直接提交总体组审批
- 总体组通过后执行删除

### 5. 拒绝回退规则

| 被拒绝的阶段 | 回退行为 |
|-------------|---------|
| 完善阶段被拒（设备负责人拒绝完善） | 整个审批请求终止，信号回退到提交前状态 |
| 审批阶段被拒（总体组拒绝） | 整个审批请求终止，信号回退到提交前状态 |

回退到提交前状态：
- 新建信号 → 回退为 Draft
- 编辑信号 → 恢复原值（从 old_payload 恢复）
- 添加端点 → 删除新增的端点
- 删除端点 → 端点保留不删

## 通知规则

| 事件 | 通知对象 | 通知类型 |
|------|---------|---------|
| 新建/编辑信号提交 | 其他端点设备负责人 | 待完善 |
| 端点完善确认完成 | 总体组成员 | 待审批 |
| 修改信号属性提交（跳过完善） | 总体组成员 | 待审批 |
| 删除端点提交 | 被删端点设备负责人 | 知会（非审批） |
| 删除端点提交 | 总体组成员 | 待审批 |
| 审批通过 | 提交人 | 审批通过 |
| 审批拒绝 | 提交人 | 审批拒绝 |
| 完善被拒 | 提交人 | 完善被拒 |
