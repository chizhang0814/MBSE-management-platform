# EICD平台审批流程逻辑梳理

> 最后更新：2026-04-13

## 一、角色权限

| 角色 | 信号操作 | 设备/连接器操作 | 针孔操作 | 审批权限 |
|------|---------|---------------|---------|---------|
| admin | 全部直接生效 | 全部直接生效 | 全部直接生效 | - |
| 总体组 | 不可操作 | 创建/编辑/删除（需审批） | - | 可审批 |
| EWIS管理员 | 创建/编辑/删除（需审批） | - | - | - |
| 系统组（设备管理员） | 创建/编辑/删除（需审批，须有自己端点） | - | 创建/编辑/删除 | - |

## 二、Completion（完善）发生的条件

Completion **只**发生在信号端点跨设备的场景中：

| 操作 | 是否有completion | 接收人 | 条件 |
|------|-----------------|--------|------|
| 创建信号 | ✅ | 所有端点的其他设备负责人 | 始终 |
| 编辑信号（端点有新增设备或换pin） | ✅ | 变化端点的设备负责人 | endpointsChanged=true且有设备变化 |
| 编辑信号（仅删除端点，保留端点不变） | ❌ | - | 跳过completion，直接approval |
| 编辑信号（仅修改信号属性，端点不变） | ❌ | - | 跳过completion，直接approval |
| 删除信号 | ❌ | - | 跳过completion，直接approval |
| 编辑针孔（有关联信号） | ✅ | 关联信号的其他端点设备负责人 | relatedSignals.length > 0 |
| 编辑针孔（无关联信号） | ❌ | - | 直接更新，无审批 |
| **其他所有操作** | ❌ | - | 无completion |

### Completion的含义

当一个操作影响了跨设备的信号端点关系时，其他设备负责人需要确认/补全自己设备端的端点信息。

### Completion如何完善（前端操作）

- **信号的completion**：对端负责人在"我的任务"里看到Pending信号，双击打开编辑端点信息，保存即完成完善（PUT信号时后端检测到有pending completion项，走完善路径）
- **针孔编辑的completion**：⚠️ 前端目前只能点"完成"确认，无法实际编辑内容

## 三、各操作详细流程

### 信号操作

#### 创建信号（POST /api/signals）

| 角色 | 流程 |
|------|------|
| admin | 直接生效（Active），无审批 |
| EWIS管理员 | 提交审批 |
| 系统组 | 提交审批（须有自己端点） |

- **提交时数据变化**：信号+端点+边全部写入DB，status=Pending
- **审批流程**：completion（对端设备负责人）→ approval（总体组，一人通过即生效）→ status=Active
- **拒绝恢复**：status→Draft，数据保留供用户修改后重新提交

#### 编辑信号（PUT /api/signals/:id）

| 角色 | 流程 |
|------|------|
| admin | 直接生效，无审批 |
| EWIS管理员/系统组 | 提交审批（视端点变化决定是否有completion） |

- **提交时数据变化**：字段+端点+边全部立即写入DB，status=Pending，旧数据保存到old_payload（含_oldEndpoints和_oldEdges）
- **审批流程**：

| 端点变化情况 | completion | approval |
|------------|-----------|---------|
| 新增设备或换pin | ✅ 变化端点的设备负责人 | 总体组 |
| 仅删除端点（保留端点不变） | ❌ 跳过 | 总体组 |
| 端点完全不变（仅改属性） | ❌ 跳过 | 总体组 |

- **拒绝恢复**：从old_payload恢复全部字段+端点+边，完整回滚到编辑前状态

#### 删除信号（DELETE /api/signals/:id）

| 角色 | 流程 |
|------|------|
| admin | 直接删除 |
| EWIS管理员/系统组（须有自己端点） | 提交审批 |

- **提交时数据变化**：仅status→Pending，数据不变
- **审批流程**：无completion，直接approval（总体组）
- **通过后**：删除信号+端点+边
- **拒绝恢复**：status恢复old_payload中的原始值

### 设备操作

#### 创建设备（POST /api/devices）

| 角色 | 流程 |
|------|------|
| admin | 直接生效（normal），无审批 |
| 总体组 | 提交审批 |

- **提交时数据变化**：设备记录写入DB，status=Pending
- **审批流程**：无completion，直接approval（其他总体组成员）
- **拒绝恢复**：status→Draft，数据保留

#### 编辑设备（PUT /api/devices/:id）

| 角色 | 流程 |
|------|------|
| admin | 直接生效 |
| 总体组 | 提交审批 |

- **提交时数据变化**：仅status→Pending，**字段不写入DB**（存在payload中，审批通过才应用）
- **审批流程**：无completion，直接approval（其他总体组成员）
- **通过后**：从payload取出字段写入设备，status→normal；若有LIN号变更则重命名连接器前缀
- **拒绝恢复**：status恢复原值（字段本来就没变，无需回滚）

#### 删除设备（DELETE /api/devices/:id）

| 角色 | 流程 |
|------|------|
| admin | 直接删除（级联删除连接器→针孔→信号端点） |
| 总体组 | 提交审批 |

- **提交时数据变化**：仅status→Pending
- **审批流程**：无completion，直接approval
- **通过后**：级联删除（连接器→针孔→信号端点，信号端点≤2则删信号，>2则仅移除端点）
- **拒绝恢复**：status恢复原值

### 连接器操作（同设备，总体组操作）

| 操作 | 提交时数据变化 | 审批流程 | 拒绝恢复 |
|------|-------------|---------|---------|
| 创建 | 记录写入，status=Pending | 仅approval（其他总体组） | status→Draft |
| 编辑 | 仅status→Pending，字段不写入 | 仅approval（其他总体组） | status恢复原值 |
| 删除 | 仅status→Pending | 仅approval（其他总体组） | status恢复原值 |

### 针孔操作

| 操作 | 条件 | 流程 |
|------|------|------|
| 创建 | 任何 | **直接生效，无审批** |
| 编辑 | 无关联信号 | **直接更新，无审批** |
| 编辑 | 有关联信号 | completion(关联信号其他设备负责人) → approval(总体组) |
| 删除 | 无关联信号 | **直接删除，无审批** |
| 删除 | 有关联信号 | 仅approval(总体组)，无completion |

针孔编辑（有关联信号）：
- 提交时仅status→Pending，字段不写入
- 拒绝时status恢复原值

## 四、审批推进逻辑

```
提交 → current_phase = 'completion'（若有completion项）或 'approval'（若无）

Completion阶段：
  所有completion项done → 自动转入Approval → 通知总体组

Approval阶段：
  一人通过 → 取消其他待审批项 → 实体生效 → 通知请求人 + 信号端点所有设备负责人
  一人拒绝 → 取消所有待处理项 → 实体回滚 → 通知请求人
```

## 五、通知

| 时机 | 接收人 | 类型 |
|------|-------|------|
| 提交审批（有completion） | completion接收人（设备负责人） | completion_request |
| Completion完成→推进到Approval | approval接收人（总体组） | approval_request |
| 审批通过 | 请求人 + 信号所有端点设备负责人 | approval_approved |
| 审批拒绝 | 请求人 | approval_rejected |
| 端点自动合并（创建信号时） | 现有信号的其他设备负责人 | signal_endpoint_added |

## 六、已知问题

1. **针孔编辑的completion无法实际编辑**：接收人只能点"完成"确认，无法编辑内容，且看到的是旧值
2. **创建针孔无审批**：系统组可直接创建
3. **编辑/删除针孔（无关联信号）无审批**：直接执行
4. **合并连接器无审批**：直接执行

## 七、数据写入时机对比

| 操作 | 字段写入时机 | 端点/子项写入时机 |
|------|-----------|---------------|
| 信号创建 | 立即写入 | 端点+边立即创建 |
| **信号编辑** | **立即写入** | **端点+边立即替换** |
| 信号删除 | 仅status | 不变 |
| 设备创建 | 立即写入 | - |
| 设备编辑 | **不写入，存payload** | 不变 |
| 设备删除 | 仅status | 不变 |
| 连接器创建 | 立即写入 | - |
| 连接器编辑 | **不写入，存payload** | 不变 |
| 连接器删除 | 仅status | 不变 |
| 针孔编辑（有关联信号） | **不写入，存payload** | 不变 |
| 针孔删除（有关联信号） | 仅status | 不变 |

> 信号编辑是唯一一个在提交审批前就写入字段和端点的操作，因此拒绝时需要全量回滚（字段+端点+边）。其他编辑操作字段存在payload中，拒绝时只需恢复status。
