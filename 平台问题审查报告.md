## 问题 #1：设备 LIN 号修改后连接器编号不联动（🔴 高危）

### 现象

用户修改设备的 `设备LIN号（DOORS）`（例如从 `8810G0000` 改为 `9999X0000`）后，该设备下所有连接器的 `设备端元器件编号` 仍然保留旧的 LIN 前缀（如 `8810G0000-SP30001`），不会随之更新为 `9999X0000-SP30001`。

### 影响

1. **连接器变成"僵尸态"**：能在列表中查看，但一旦尝试编辑，后端校验函数 `validateConnectorCompId()` 会要求编号以新 LIN 号开头，导致校验失败、无法保存。
2. **新建连接器格式冲突**：前端创建新连接器时自动以新 LIN 号为前缀，与旧连接器格式不一致。
3. **数据导出不一致**：导出的 EICD 文档中，设备 LIN 号与连接器编号不匹配。

### 根因分析

**涉及文件：**
- `server/src/routes/devices.ts` 第 632-728 行（PUT /api/devices/:id 路由）
- `server/src/shared/column-schema.ts` 第 248-279 行（validateConnectorCompId 校验函数）

**问题链路：**

```
用户修改设备LIN号
  → PUT /api/devices/:id
  → 第 662-676 行：校验新LIN号唯一性（通过）
  → 第 690-696 行：UPDATE devices 表（成功）
  → ❌ 无任何逻辑更新 connectors 表
  → connectors.设备端元器件编号 仍为旧值
  → 后续编辑连接器时 validateConnectorCompId() 校验失败
```

**数据库层面：**
- `connectors` 表的 `设备端元器件编号` 字段中嵌入了设备 LIN 号作为前缀（如 `{LIN}-SP30001`），这是冗余的业务编码约定。
- `connectors` 表对 `devices` 表仅有 `device_id` 外键（`ON DELETE CASCADE`），无 UPDATE 级联。
- SQLite 不支持 `ON UPDATE CASCADE` 触发器自动更新子表非外键字段。

### 修改方法

在 `server/src/routes/devices.ts` 的 PUT /api/devices/:id 路由中，在更新 devices 表之后、返回响应之前，增加级联更新逻辑：

```typescript
// 在第 690 行 UPDATE devices 之后添加：

// 1. 检测 LIN 号是否实际变更
const oldLin = String(device['设备LIN号（DOORS）'] || '').trim();
const newLin = String(fields['设备LIN号（DOORS）'] || oldLin).trim();
const linChanged = newLin !== oldLin;

// 2. 如果变更，用事务原子性地更新所有关联连接器
if (linChanged) {
  await db.run('BEGIN TRANSACTION');
  try {
    const connectors = await db.query(
      `SELECT id, "设备端元器件编号" as compId FROM connectors WHERE device_id = ?`,
      [deviceId]
    );
    const oldPrefix = oldLin + '-';
    for (const conn of connectors) {
      const oldCompId = String(conn.compId || '');
      if (oldCompId.startsWith(oldPrefix)) {
        const newCompId = newLin + '-' + oldCompId.slice(oldPrefix.length);
        await db.run(
          `UPDATE connectors SET "设备端元器件编号" = ?, updated_at = CURRENT_TIMESTAMP WHERE id = ?`,
          [newCompId, conn.id]
        );
      }
    }
    // 记录级联更新日志到 change_logs
    await db.run('COMMIT');
  } catch (err) {
    await db.run('ROLLBACK');
    throw err;
  }
}
```

**前端联动：** 在 `client/src/pages/ProjectDataView.tsx` 的 `saveDevice` 函数中（第 781 行），根据后端返回的 `cascadedConnectors` 数量提示用户。

**注意事项：**
- 必须使用事务包裹：如果中途某个连接器更新失败（如唯一约束冲突），需要整体回滚。
- 应记录变更日志到 `change_logs` 表，便于审计追溯。
- 需考虑连接器处于 `Pending`（审批中）状态时的处理策略——建议禁止在有 Pending 连接器时修改 LIN 号，或在级联更新后同时更新审批请求中的快照数据。

## 问题 #2：多步骤数据操作普遍缺乏事务保护（🔴 高危）

### 现象

平台中大量涉及多个表的写操作没有使用数据库事务（`BEGIN TRANSACTION` / `COMMIT` / `ROLLBACK`），如果中途因任何原因失败（网络、并发冲突、约束违反），会导致数据处于不一致的中间状态。

### 影响

以设备级联删除为例：删除一个设备需要依次删除其关联的信号端点、针孔、连接器、最后删除设备本身。如果在删除针孔后、删除连接器前服务崩溃，数据库中会出现没有针孔的连接器"空壳"。

### 涉及位置

| 操作 | 文件 | 行号 | 涉及的表 |
|------|------|------|---------|
| 设备级联删除 | `devices.ts` | 16-150 | signal_endpoints → signals → pins → connectors → devices |
| 连接器级联删除 | `devices.ts` | 111-130 | pins → connectors（通过 `cascadeDeleteConnector`） |
| 针孔级联删除 | `approval-helper.ts` | 68-112 | signal_endpoints → signals → pins（通过 `cascadeDeletePinShared`） |
| 信号编辑（含端点重建） | `signals.ts` | 667-760 | signal_endpoints（删除+重建）→ pins（更新属性）→ signal_edges（重建） |
| 审批通过后更新实体 | `approval-helper.ts` | 280-340 | approval_requests → approval_items → devices/connectors/pins/signals |
| 项目全部设备删除 | `devices.ts` | 731-740 | signal_endpoints → devices（CASCADE 删除连接器和针孔） |

**对比：** 项目复制操作（`projects.ts` 第 357-540 行）**正确地使用了事务**，可作为修复参考模式。

### 修改方法

为每个涉及多表写操作的路由/函数添加事务包裹。以设备级联删除为例：

```typescript
// devices.ts — cascadeDeleteDevice 函数改造
async function cascadeDeleteDevice(db: Database, deviceId: number, userId: number, log: string[]) {
  await db.run('BEGIN TRANSACTION');
  try {
    // ... 原有级联删除逻辑 ...
    await db.run('COMMIT');
  } catch (err) {
    await db.run('ROLLBACK');
    throw err;
  }
}
```

**建议封装一个事务辅助函数，统一使用：**

```typescript
// database.ts 或 utils 中添加
async function withTransaction<T>(db: Database, fn: () => Promise<T>): Promise<T> {
  await db.run('BEGIN TRANSACTION');
  try {
    const result = await fn();
    await db.run('COMMIT');
    return result;
  } catch (err) {
    await db.run('ROLLBACK');
    throw err;
  }
}
```

**需要改造的函数清单：**
1. `cascadeDeleteDevice()`（`devices.ts:16`）
2. `cascadeDeleteConnector()`（`devices.ts:111`）
3. `cascadeDeletePinShared()`（`approval-helper.ts:68`）
4. `PUT /api/signals/:id` 的端点重建分支（`signals.ts:667`）
5. `POST /api/devices/:devId/connectors/merge`（`devices.ts:1004`）
6. `checkAndAdvancePhase()`（`approval-helper.ts` 审批阶段推进）

## 问题 #3：signal_endpoints 外键策略混用（🟡 中危）

### 现象

`signal_endpoints` 表有两个外键，使用了不同的删除策略：

```sql
-- database.ts 第 242-252 行
CREATE TABLE signal_endpoints (
  ...
  FOREIGN KEY (signal_id) REFERENCES signals(id) ON DELETE CASCADE,   -- 信号删除时自动删除
  FOREIGN KEY (pin_id)    REFERENCES pins(id)    ON DELETE RESTRICT   -- 针孔被引用时禁止删除
);
```

但在应用代码中，部分地方绕过了 `RESTRICT` 约束：

- `devices.ts` 第 735-739 行：删除项目所有设备时，**先手动删除** `signal_endpoints`，然后才删设备（触发 CASCADE 删设备→连接器→针孔）。
- `approval-helper.ts` 第 92-101 行：`cascadeDeletePinShared` 中，先手动删除该 pin 关联的 `signal_endpoints`，再删除 pin。

而另一些地方直接依赖外键约束，如果有信号引用了某个针孔，删除针孔会直接报外键约束错误。

### 影响

- 不同的删除入口行为不一致：有的地方手动清理后可以删除，有的地方直接报错。
- 代码维护困难：开发者需要记住哪些地方需要手动清理、哪些不需要。

### 修改方法

**方案 A（推荐）：统一为应用层手动清理**

将所有删除操作改为：先查询关联的 signal_endpoints，按逻辑处理（信号端点 ≤ 2 个时整体删除信号，> 2 个时仅删除该端点），再删除 pin。`cascadeDeletePinShared` 已经实现了这个逻辑，确保所有删除 pin 的入口都调用该函数即可。

**方案 B：将外键改为 CASCADE**

```sql
FOREIGN KEY (pin_id) REFERENCES pins(id) ON DELETE CASCADE
```

但这会导致删除针孔时静默删除信号端点，可能丢失用户不期望丢失的信号数据。不推荐。

**建议：** 采用方案 A，并在所有删除针孔的代码路径中，统一通过 `cascadeDeletePinShared()` 执行，不直接 `DELETE FROM pins`。

## 问题 #4：信号编辑时 pins 属性非原子更新（🟡 中危）

### 现象

编辑信号（`PUT /api/signals/:id`）时，如果请求体包含 `endpoints` 数组，后端会执行以下多步操作：

```
第 679 行：DELETE FROM signal_endpoints WHERE signal_id = ?   （删除所有旧端点，CASCADE 删除 edges）
第 685-719 行：FOR EACH endpoint:
    → INSERT INTO signal_endpoints（创建新端点）
    → UPDATE pins SET 端接尺寸=?, 屏蔽类型=?（更新针孔属性）
第 723-746 行：INSERT INTO signal_edges（重建连接关系）
```

这 3 个步骤共涉及 3 张表（`signal_endpoints`、`pins`、`signal_edges`），但没有事务包裹。

### 影响

- 如果在第 679 行删除旧端点后、创建新端点前服务崩溃，信号会变成没有任何端点的"空壳"。
- 如果某个端点创建成功但对应的 pin 属性更新失败，`pins.端接尺寸` 与 `signal_endpoints` 中的值不一致。
- 如果 edges 重建部分失败，信号的连接拓扑不完整。

### 涉及代码

`server/src/routes/signals.ts` 第 667-760 行。

### 修改方法

用事务包裹整个端点重建流程：

```typescript
// signals.ts 第 667 行
if (Array.isArray(endpoints)) {
  await db.run('BEGIN TRANSACTION');
  try {
    // 1. 保存旧 edges
    const oldEdges = await db.query(...);
    
    // 2. 删除旧端点（CASCADE 删除 edges）
    await db.run('DELETE FROM signal_endpoints WHERE signal_id = ?', [signalId]);
    
    // 3. 创建新端点 + 更新 pin 属性
    for (const ep of endpoints) {
      await db.run('INSERT INTO signal_endpoints ...', [...]);
      if (pinId) await db.run('UPDATE pins SET ...', [...]);
    }
    
    // 4. 重建 edges
    for (const oe of oldEdges) {
      await db.run('INSERT INTO signal_edges ...', [...]);
    }
    
    await db.run('COMMIT');
  } catch (err) {
    await db.run('ROLLBACK');
    throw err;
  }
}
```

## 问题 #5：Pending 状态信号可被重新编辑（🟡 中危）

### 现象

信号处于 `Pending`（审批中）状态时，`PUT /api/signals/:id` 路由仍然允许编辑。编辑会删除并重建所有 `signal_endpoints` 和 `signal_edges`。

### 涉及代码

`server/src/routes/signals.ts` 第 618-760 行。路由入口处（第 618-665 行）仅检查了用户权限，未检查信号状态。

### 影响

- **审批人看到的数据与最终生效的数据不一致**：审批人基于提交时的快照进行审批，但信号内容可能在审批期间被修改。
- **审批项失效**：审批项（`approval_items`）关联的是信号 ID，但信号内容已经变了。
- **端点关系错乱**：重建端点后 pin_id 映射改变，审批通过时执行的后续操作可能基于过期数据。

### 修改方法

**方案 A（推荐）：禁止编辑 Pending 信号**

在 PUT 路由入口添加状态检查：

```typescript
// signals.ts 第 622 行后添加
if (signal.status === 'Pending') {
  return res.status(400).json({ error: '审批中的信号无法编辑，请等待审批完成或撤回后再修改' });
}
```

**方案 B：允许编辑但自动撤回审批**

编辑 Pending 信号时，自动将关联的审批请求标记为 `withdrawn`（撤回），信号状态回退为 `Draft`，用户需重新提交审批。

## 问题 #6：审批通过后校验失败静默回退 Draft，不通知用户（🟢 低危）

### 现象

审批通过后，`approval-helper.ts` 中的 `revalidateDevice()` 函数（第 114-167 行）会重新校验设备字段。如果校验失败（例如设备编号与 `aircraft_device_list` 不匹配），设备的 `validation_errors` 会被更新，但 `status` 会从 `normal` 回退为 `Draft`——**且不发送任何通知**。

### 涉及代码

- `server/src/shared/approval-helper.ts` 第 114-167 行（`revalidateDevice` 函数）
- 调用点：`approval-helper.ts` 第 339 行（审批通过后调用）
- 调用点：`devices.ts` 第 708-719 行（设备编辑后调用）

### 影响

用户提交设备 → 审批人通过 → 设备看起来应该是 `normal` 状态 → 实际上因为某个字段校验失败又变回了 `Draft` → 用户和审批人都不知道。

### 修改方法

在 `revalidateDevice()` 检测到校验失败时，向设备负责人和提交审批的用户发送通知：

```typescript
// approval-helper.ts revalidateDevice 函数末尾
if (veErrors.length > 0) {
  await db.run(`UPDATE devices SET status = 'Draft', validation_errors = ? WHERE id = ?`,
    [JSON.stringify(veErrors), deviceId]);
  
  // 发送通知
  await db.run(
    `INSERT INTO notifications (recipient, type, title, content, reference_type, reference_id)
     VALUES (?, 'warning', ?, ?, 'device', ?)`,
    [device['设备负责人'],
     '设备校验未通过',
     `设备 ${device['设备编号']} 审批通过后重新校验发现问题：${veErrors.join('、')}，已回退为草稿状态`,
     deviceId]
  );
}
```

## 问题 #7：虚拟字段依赖前端手动删除（🟢 低危）

### 现象

后端查询设备/连接器列表时，会通过 SQL JOIN 或子查询附加一些虚拟字段（不存在于数据库表中的计算字段）。前端在编辑保存时，必须手动 `delete` 这些字段，否则后端 SQL 会报 `SQLITE_ERROR: no such column`。

### 涉及虚拟字段

| 字段名 | 来源 | 使用场景 |
|--------|------|---------|
| `has_pending_sub` | 设备列表查询动态附加 | 表示子实体有待审批项 |
| `pending_item_type` | 查询时 JOIN approval_items | 当前用户的待处理类型 |
| `pending_sub_item_type` | 查询时动态附加 | 子实体的待处理类型 |
| `connector_count` | 查询时 COUNT 子查询 | 连接器数量统计 |
| `pin_count` | 查询时 COUNT 子查询 | 针孔数量统计 |
| `设备负责人姓名` | 查询时 JOIN users | 负责人显示名 |
| `management_claim_requester` | 查询时动态附加 | 管理权申请人 |

### 涉及代码

- 清理位置：`devices.ts` 第 652-660 行（PUT 设备时手动 delete）
- 清理位置：`devices.ts` 第 873 行（PUT 连接器时手动 delete）

### 影响

- 如果前端或后端遗漏了某个虚拟字段的清理，保存操作会直接报错。
- 每次新增虚拟字段都需要同步更新前后端的清理逻辑，容易遗漏。

### 修改方法

在后端 PUT 路由中，改为白名单机制——只取数据库表中实际存在的列名，忽略其他字段：

```typescript
// 通用方法：根据表的 PRAGMA 信息过滤字段
async function filterToDbColumns(db: Database, tableName: string, fields: Record<string, any>): Record<string, any> {
  const columns = await db.query(`PRAGMA table_info("${tableName}")`);
  const colNames = new Set(columns.map((c: any) => c.name));
  // 排除自动管理的列
  const excluded = new Set(['id', 'created_at', 'updated_at', 'version', 'status']);
  return Object.fromEntries(
    Object.entries(fields).filter(([k]) => colNames.has(k) && !excluded.has(k))
  );
}

// PUT /api/devices/:id 中使用
const safeFields = await filterToDbColumns(db, 'devices', fields);
```

## 问题 #8：连接器合并操作无事务回滚（🟢 低危）

### 现象

合并连接器（`POST /api/devices/:devId/connectors/merge`，`devices.ts` 第 1004-1078 行）的流程是：
1. 遍历每个源连接器
2. 将源连接器下的所有针孔移动到目标连接器（`UPDATE pins SET connector_id = ?, 针孔号 = ?`）
3. 删除空的源连接器

如果在步骤 2 中，某个针孔重命名后违反唯一约束（`UNIQUE(connector_id, 针孔号)`），已经成功移动的针孔不会回滚。

### 涉及代码

`server/src/routes/devices.ts` 第 1030-1064 行。

### 影响

- 部分针孔被移动到目标连接器、部分仍在源连接器——数据处于不可预期的中间状态。
- 源连接器可能已被删除但其针孔还留在原位（因为删除在移动之后）。

### 修改方法

```typescript
// devices.ts 第 1029 行之后
await db.run('BEGIN TRANSACTION');
try {
  for (const sourceId of sourceConnectorIds) {
    // ... 原有迁移逻辑 ...
  }
  await db.run('COMMIT');
} catch (err) {
  await db.run('ROLLBACK');
  return res.status(500).json({ error: `合并失败已回滚：${err.message}` });
}
```

## 问题 #9：信号 - 针孔跨表编辑的乐观锁盲区（🟢 低危）

### 现象

设备、连接器、针孔、信号各自有 `version` 字段实现乐观锁：

```sql
UPDATE devices SET ..., version = version + 1 WHERE id = ? AND version = ?
```

但信号编辑（`PUT /api/signals/:id`）时会跨表更新 pins 的属性（`signals.ts` 第 711-718 行）：

```typescript
await db.run(`UPDATE pins SET "端接尺寸" = ?, "屏蔽类型" = ? WHERE id = ?`, [...]);
```

这个 UPDATE 没有检查 pin 的 version，也不会递增 pin 的 version。

### 影响

**并发场景：**
- 用户 A 通过信号编辑页修改了针孔的 `端接尺寸`（走 signals 路由，不检查 pin version）
- 用户 B 同时通过针孔编辑页修改了同一个针孔的 `端接尺寸`（走 devices 路由，检查 pin version）
- 两个操作都会成功，后执行的覆盖先执行的，没有冲突检测。

### 涉及代码

- 写入方：`signals.ts` 第 711-718 行（信号编辑时更新 pin 属性，不检查 version）
- 读取方：`devices.ts` 第 1141 行（针孔编辑时检查 version）

### 修改方法

**方案 A（轻量级）：信号编辑时也递增 pin 的 version**

```typescript
// signals.ts 第 717 行改为
await db.run(
  `UPDATE pins SET ${pinUpdates.join(', ')}, version = version + 1, updated_at = CURRENT_TIMESTAMP WHERE id = ?`,
  [...pinVals, pinId]
);
```

这不能完全防止并发覆盖，但至少让后续的针孔编辑能检测到 version 变化。

**方案 B（完整方案）：信号编辑时检查并锁定 pin 的 version**

在信号编辑请求中携带各端点 pin 的当前 version，后端 UPDATE 时加 `AND version = ?` 条件。但这需要前端额外传递 pin version 数据，改动较大。

**建议：** 当前并发编辑频率不高的情况下，方案 A 即可。后续如果出现实际冲突，再升级为方案 B。
